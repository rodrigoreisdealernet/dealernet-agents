  • Every stage is wrapped in timeout N + warn-and-continue — one stuck agent never blocks the pass. 330 s ≈ the SDK's idle-timeout plus buffer; the Tech Reviewer gets 20 min because deep multi-file review legitimately runs long.
  • Lanes run before the Tech Reviewer so an open needs-* lane is cleared and the PR approved in the same pass.
  • This pass never merges or assigns. It produces triage and approvals; the PR Loop (its own workflow, every 30 min) consumes them.
  • Timer-only is load-bearing: an event trigger once caused self-cancellation (69% of runs died mid-pass). The design is pinned by a contract test in temporal/tests/.

  • The agent is a markdown file. The body becomes the system prompt; {{max_open_copilot_prs}}-style variables are filled from factory.yml. Changing behaviour = a one-file docs PR.
  • Full autonomy, bounded blast radius. Permission requests are auto-approved — the workflow's permissions: block and the tokens are the real security boundary.
  • Two tokens, two jobs: COPILOT_TOKEN authenticates the model session (the brain); PROJECT_MANAGER_PAT backs every gh command (the hands).
  • Timeouts are owned by the caller, not the prompt: pipeline-fast wraps stages in timeout 330; the PR Loop gives each session 8 min and aborts it on overrun.

Why the SDK and not a plain LLM API call? A chat completion has no tool access — an early Azure-REST swap was reverted for exactly that reason.

E2E is out-of-band — by design
  • Playwright tests the deployed dev app, not the PR branch — hourly and after every deploy
  • Gating smoke specs file deduped incidents on failure
  • An "experience" suite is intentionally non-gating: desired-UX expectations that act as a backlog signal, never a deploy blocker
  • The QA Manager agent reads the E2E history to spot regressions and flaky specs
Trust boundaries

What stays human-gated

The factory does autonomously
  • Triage, design, assignment, implementation
  • Code review with terminal verdicts — reviewers may not punt; every PR reaches approve or request-changes
  • Merging to main — the human merge gate was removed once reviewer decisions became terminal
  • Dev deploys, E2E, incident filing, docs upkeep
Only humans can
  • Promote to test — manual dispatch with an explicit build run id
  • Deploy to prod — protected-environment reviewer approval
  • Mutate the cluster — remediation needs manual dispatch and an environment approval; a schedule can never do it
  • Fix account-level auth (billing entity, token types) — agents can detect these, not repair them

The pattern: humans gate state changes in the world (prod, cluster, money); agents own everything that is reversible by another merge.
